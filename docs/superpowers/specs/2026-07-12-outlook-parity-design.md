# Outlook feature parity with Gmail — design

Date: 2026-07-12
Status: approved for implementation (user pre-approved self-review flow)

## Goal

A user who connects Outlook gets the same "keep using your own inbox" experience a
Gmail user gets today. FlowDesk works directly inside their Outlook mailbox:

- Auto-labeling: FlowDesk labels projected as Outlook **categories** on messages.
- Bulk archive / bulk unsubscribe from Clean Inbox (and single-conversation
  archive/trash/read actions).
- AI reply drafts created in (and withdrawn from) the user's Drafts folder.
- Rules: already provider-neutral (they set `attentionCategory`); their effects
  must reach Outlook through the same label-projection path Gmail uses.
- Learning from user edits: when the user adds/removes FlowDesk categories in
  Outlook, record classification corrections (same as Gmail label feedback).
- Drift reconciliation (read-state + labels), operator health, onboarding
  first-pass, and a "Fix labels" catch-up — all at parity.

## Current state (from code exploration)

Ingest side is already built and tested: OAuth (`lib/microsoft.ts`), leased delta
sync (`lib/outlook-sync.ts`), Graph change-notification subscriptions + webhook
queue + worker (`lib/outlook-subscriptions.ts`, `lib/outlook-notifications.ts`,
`lib/outlook-worker.ts`), send/reply with CC/BCC (`lib/conversations/send-message.ts`).

Everything action/writeback-shaped is Gmail-only:

- `lib/agent/gmail-writeback-processor.ts:91` skips any non-`google` channel.
- `lib/gmail-labels.ts:283` projection returns null for non-`google`.
- `lib/clean-inbox-gmail.ts:18` excludes Outlook conversations from bulk archive.
- `app/api/conversations/[id]/{archive,trash,read,status}` guard `provider !== "google"`.
- No Outlook drafts, feedback learning, state/label reconcile, operator health,
  relabel, or onboarding first-pass.
- `docs/TODO.md:90–91` already tracks "Outlook archive/trash/unsubscribe parity"
  and "record Outlook renewal/sync failure causes instead of bare catch blocks".

Classification, rules, unsubscribe-URL parsing, sender cleanup grouping, and the
automation-level gates are provider-neutral already. `Mail.ReadWrite` scope is
already requested, so existing connections don't need re-consent.

## Approaches considered

**A. Mirror the Gmail stack file-for-file** (outlook-writeback-queue model,
outlook-writeback-processor, outlook clean-inbox, …). Fastest to write, but
duplicates all orchestration (queue claim/backoff/audit/echo-suppression/level
gating) that then has to be kept in sync twice. Rejected: violates the DRY goal.

**B. Provider adapter for writeback (chosen).** Keep per-provider API modules
thin and concrete (`lib/google.ts` stays; new `lib/outlook-mailbox.ts` for Graph
mailbox ops), and generalize the orchestration layer — one queue model, one
writeback processor, one projection path, one clean-inbox module, shared
reconcile cores — dispatching through a small adapter interface keyed by
`Channel.provider`. Sync stays per-provider (history vs delta are legitimately
different, both already work).

**C. Full `EmailProvider` abstraction including sync/auth/send.** Cleanest on
paper but rewrites working, tested sync code with no user-visible gain.
Rejected: unnecessary churn.

## Design

### Concept mapping (Gmail → Outlook / Microsoft Graph)

| FlowDesk concept | Gmail | Outlook (Graph) |
|---|---|---|
| Labels | user labels; `threads.modify` | master categories (`/me/outlook/masterCategories`) + `PATCH /me/messages/{id}` `categories` per message |
| Archive | remove `INBOX` label | move message to `archive` well-known folder |
| Restore (undo) | add `INBOX` | move back to `inbox` |
| Trash | `threads.trash` | move to `deleteditems` |
| Mark read | remove `UNREAD` | `PATCH` `isRead: true` per message |
| Draft reply | `drafts.create` with `threadId` | `POST /me/messages/{id}/createReply` then `PATCH` body (stays in Drafts) |
| Withdraw draft | `drafts.delete` | `DELETE /me/messages/{draftId}` |
| User label edits → feedback | `history.list` labelsAdded/Removed | delta sync: add `categories`, `isRead` to `$select`; diff against expected |
| Operation scope | thread-level | message-level; apply to the conversation's stored messages, bounded (most recent N=20) |

Category names are the existing flat canonical names (`Needs Reply`, `Handled`,
…) from the single taxonomy source. Colors map from the Gmail palette to the
nearest Graph preset color (`preset0`–`preset24`). If the user already has a
category with a matching name, adopt it (same as Gmail's get-or-create).

### Schema changes (one migration)

1. **Rename `GmailWritebackQueue` → `EmailWritebackQueue`** (relations:
   `Channel.emailWritebackQueue`, `Conversation.emailWritebacks`, tenant
   relation likewise). Hand-written `ALTER TABLE … RENAME TO …` migration so
   data is preserved (Prisma's autogenerated diff would drop+create). Columns
   and action strings (`apply_labels`, `mark_read`, `create_draft`,
   `withdraw_draft`) unchanged; rows are provider-scoped via `channelId`.
2. **No new models.** `OutlookCredential` already has subscription/delta/lease
   fields. `GmailLabelMapping` is reused as the provider-neutral FlowDesk label
   taxonomy config (it stores canonical-name enable/disable per tenant — nothing
   Gmail-specific but the name; renaming it is peripheral churn, deferred, doc
   comment updated instead).
3. **No new Message columns.** Outlook feedback diffs delta-reported categories
   against the categories expected from `ConversationState`, with echo
   suppression via recent `EmailWritebackQueue` rows (same strategy as Gmail's
   feedback echo consumption). The existing `metadataJson.gmailLabelOverride`
   key is reused as the override marker for both providers (no data migration;
   helper names generalized).

### Module layout

New:
- `lib/email/writeback-adapter.ts` — `EmailWritebackAdapter` interface +
  `getWritebackAdapter(provider)`: `ensureLabels`, `applyLabels`,
  `markConversationRead`, `archiveConversation`, `restoreConversation`,
  `trashConversation`, `createDraft`, `deleteDraft`. Gmail impl delegates to
  existing `lib/google.ts` functions; Outlook impl to `lib/outlook-mailbox.ts`.
  Unsupported providers (twilio) return null → callers keep current skip
  behavior.
- `lib/outlook-mailbox.ts` — concrete Graph ops per the mapping table, built on
  `graphRequest`/`getOutlookAccessToken` from `lib/microsoft.ts`. Preserves the
  user's non-FlowDesk categories when patching. Bounded message fan-out.
- `lib/agent/label-feedback-core.ts` — provider-neutral correction logic
  extracted from `gmail-label-feedback.ts`; consumed by both the Gmail history
  path and a new Outlook delta path (`lib/agent/outlook-category-feedback.ts`).

Generalized (renamed where the file is orchestration, kept where provider-specific):
- `lib/agent/gmail-writeback-processor.ts` → `lib/agent/email-writeback-processor.ts`:
  provider gate becomes adapter lookup; audit actions become
  `{gmail|outlook}.writeback.{completed|failed}` (Gmail names unchanged for
  continuity); queue insert audit likewise `{gmail|outlook}.labels.queued`.
- Label projection in `lib/gmail-labels.ts`: the taxonomy + state→labels mapping
  is already provider-neutral; the projection entry point accepts
  `google | microsoft` and routes the writeback through the queue exactly as
  today. (File may be split into `lib/email/label-projection.ts` + thin Gmail
  exports — decided in the plan; behavior identical for Gmail.)
- `lib/clean-inbox-gmail.ts` → `lib/clean-inbox-email.ts`: archivable = provider
  in {google, microsoft} with an external thread id; archive/restore via
  adapter. Batch routes (`archive-batch`, `unsubscribe-batch`, `undo`) updated.
- Conversation action routes (`archive`, `trash`, `read`, `status`,
  `workflow-status`, `draft/suggest`): replace `provider !== "google"` guards
  with adapter capability checks.
- Drafts: queueing in `lib/gmail-drafts.ts` is already mostly neutral; draft
  metadata gains a neutral `providerDraftId` (legacy `gmailDraftId` still read).
  Automation-level keys `apply_gmail_labels` / `create_gmail_drafts` are reused
  for Outlook (they mean "in your mailbox"); copy updated, keys unchanged.
- Reconcile crons: `gmail-state-reconcile` and `gmail-label-reconcile` cores
  become provider-aware (iterate email channels of both providers; ensureLabels
  + re-project via adapter). Scheduler registry entries renamed to
  `email-writeback`, `email-state-reconcile`, `email-label-reconcile`; cron HTTP
  routes keep their existing paths and stay `CRON_SECRET`-gated, plus new
  outlook-relevant behavior rides the existing routes.
- Operator health: `lib/gmail-operator-health.ts` pattern extended with an
  Outlook summary (`auth/sync staleness`, `subscription renewal errors`,
  `writeback queue` for microsoft channels); settings page assembles inputs and
  renders health for whichever providers are connected.
- Relabel: shared catch-up handler; `POST /api/connectors/outlook/relabel` added
  alongside the Gmail route; "Fix labels" button shown for Outlook channels.
- Onboarding: first-pass generalized to run for the connected channel regardless
  of provider (classification is neutral; labels flow through projection).
  Outlook callback redirects new connections to `/onboarding?connected=outlook`;
  wizard offers Outlook when `MICROSOFT_CLIENT_ID/SECRET` configured and the
  existing capability gating allows; copy adapts ("categories in Outlook").
- Outlook worker (`lib/outlook-worker.ts`): record failure causes into
  `subscriptionError` / `lastSyncError` / audit instead of bare `catch { errors++ }`,
  and stop counting failed channels as processed (closes `docs/TODO.md:90`).

Sync-side addition (small): `lib/outlook-sync.ts` `$select` gains `categories`
and `isRead`; updated messages feed the feedback module and keep the
conversation unread flag in sync.

### Data flow (Outlook, end-to-end)

1. Mail arrives → Graph notification → `OutlookSyncEvent` → worker → delta sync
   upserts Conversation/Message (existing).
2. `syncConversationWorkItems` classifies (existing, neutral) →
   `projectFlowDeskLabelsForConversation` now proceeds for `microsoft` →
   enabled-label filtering (`GmailLabelMapping`) + automation-level gate →
   upsert `EmailWritebackQueue(apply_labels)` → inline drain → adapter →
   `outlook-mailbox` ensures master categories and PATCHes message categories →
   audit `outlook.writeback.completed`.
3. User edits categories in Outlook → delta reports changed `categories` →
   feedback module consumes writeback echoes, records
   `ClassificationCorrection` + override marker, updates state → audit
   `outlook.labels.corrected`.
4. Clean Inbox archive/unsubscribe batches include microsoft conversations →
   adapter moves messages to Archive; undo moves them back.
5. Draft approval at automation level ≥3 queues `create_draft` → adapter creates
   a Graph reply draft; new inbound or send withdraws it.

### Error handling

- Graph 401/invalid_grant → surface as connection-needs-reauth on the channel
  (`lastSyncError`, operator health), same UX as Gmail's dead-token path.
- Graph 404 on message ops → treat as skip (message moved/deleted by user).
- 429/5xx → existing queue backoff (`nextWritebackAttemptDate`, max 3 attempts,
  terminal audit `outlook.writeback.failed`).
- Partial category application (some messages patched, some failed) → job
  retries; PATCH is idempotent.

### Testing

Vitest, mirroring the existing per-feature Gmail suites with the established
inline Graph-mock pattern from `tests/outlook-*.test.ts`:

- outlook category projection + enabled-label filtering + automation gating
- generalized writeback processor dispatch (google unchanged, microsoft ops,
  twilio skipped) + audit names
- archive/trash/restore/read ops incl. clean-inbox batch inclusion and undo
- drafts create/withdraw incl. metadata compatibility (`gmailDraftId` legacy)
- category feedback: user add/remove, echo suppression, override marker
- reconcile crons over mixed-provider channels
- operator health summaries
- route guards: action routes accept microsoft, still reject twilio
- regression: full existing Gmail + Outlook suites pass after the queue rename

### Non-goals

- No rewrite of sync/auth/send layers; no unified `EmailProvider` interface.
- No rename of `GmailLabelMapping`, `gmail-label-settings` route path, or
  `Conversation.gmailUnread`/`gmailRawState` columns (peripheral naming churn;
  code comments note they now serve both providers).
- No Outlook subscriptions for folders beyond Inbox.
- Exact color fidelity for categories (nearest preset only).

### Risks

- **Queue rename migration**: must be a hand-written RENAME; verified against a
  dev database and by the full test suite. Rows in flight keep working (columns
  unchanged).
- **Category fan-out**: Outlook ops are per-message; bounded to the most recent
  20 stored messages per conversation to cap Graph calls.
- **Echo loops**: our own PATCHes come back through delta; suppressed via recent
  writeback rows (same proven approach as Gmail).
- **Global category namespace**: categories are account-wide; name-match
  adoption avoids duplicates, and we never delete user categories.
