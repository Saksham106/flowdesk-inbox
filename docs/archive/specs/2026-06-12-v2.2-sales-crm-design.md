## Overview

Two independent slices that share no new DB models:

1. **Sales agent mode** — classify sales signals in every synced conversation, surface them in a SalesPanel on conversation pages and a filter + count chip in the inbox.
2. **Mini CRM pipeline reporting** — score-range and stage filters on `/leads`, a pipeline summary bar, and a week-over-week leads table.

---

## Architecture

### Branch strategy

Create `feat/v2.2-sales-crm` from `feat/v2.1-kb-support-mode` (NOT main — v2.1 PR #24 is open as of 2026-06-12). v2.2 touches the same files as v2.1 so stacking avoids conflicts. When PR #24 merges, rebase v2.2 branch onto main.

### No new DB models

Sales signals write into the existing `ConversationState.metadataJson` JSON blob — same slot support classification uses. Pipeline reporting is computed from existing `Lead` fields (`stage`, `score`, `estimatedValue`, `createdAt`, `scoredAt`). No migrations needed.

## Feature 1: Sales Agent Mode

### `lib/agent/sales-classifier.ts`

Pure function, regex-based, mirroring `support-classifier.ts` exactly.

**Return type:**
```ts
export type SalesSignals = {
  isSalesLead: boolean
  extractedBudget: string | null
  extractedTimeline: string | null
  closingStage: "prospect" | "qualified" | "proposal" | "closing" | null
  suggestedAction: string
}
```

**Signal detection (regex patterns):**
- Budget: dollar amounts, "budget", "pricing", "cost", "quote", "investment"
- Timeline: "by [date]", "deadline", "ASAP", "urgent", "this week/month/quarter"
- Decision-maker: "our team has decided", "we'd like to move forward", "can you send a contract", "ready to sign"
- Closing-stage keywords: "ready to sign", "send contract", "move forward", "purchase order"
- Proposal language: "proposal", "quote", "scope of work", "pricing breakdown"
- Qualification signals: "budget approved", "decision maker", "evaluating", "shortlisted"

**Stage inference logic (highest signal wins):**
1. Closing-stage keywords → `"closing"`
2. Proposal language → `"proposal"`
3. Qualification signals → `"qualified"`
4. Budget mention only → `"prospect"`
5. None of the above → `isSalesLead: false`, `closingStage: null`, `suggestedAction: ""`

**Static closing-action map by stage:**
- `"prospect"` → `"Send intro deck and ask about their timeline"`
- `"qualified"` → `"Schedule a discovery call to confirm budget and requirements"`
- `"proposal"` → `"Follow up on the proposal and offer to answer questions"`
- `"closing"` → `"Send the contract and confirm next steps"`

**Exported function signature:**
```ts
export function classifySalesSignals(
  messages: { direction: string; body: string }[]
): SalesSignals
```

### `lib/agent/work-item-sync.ts`

Call `classifySalesSignals()` after `classifySupportSignals()`, before the function returns.

- Write `isSalesLead`, `extractedBudget`, `extractedTimeline`, `closingStage` to `ConversationState.metadataJson` only when `isSalesLead === true` (same guard pattern as support).
- Audit-log action: `"conversation_state.sales_classified"`.
- Return shape extended: `{ ..., salesClassified: boolean }`.

### `lib/agent/command-center.ts`

- Add `"sales_qualified"` to `CommandCenterState` union.
- New `isSalesQualified()` helper: reads `metadataJson.isSalesLead === true`.
- Add `salesQualified: number` to `counts` in `DailyCommandCenter`.
- Add `salesQualified: CommandCenterConversation[]` to `groups`.
- Score bonus: `+35` (above support's `+30`; sales-qualified is high-value and actionable).

## Feature 2: Mini CRM Pipeline Reporting

### `/leads` page — URL param filters

Parsed server-side from `searchParams` in the page server component.

| Param | Type | Prisma clause |
|---|---|---|
| `minScore` | number | `score: { gte: minScore }` |
| `maxScore` | number | `score: { lte: maxScore }` |
| `stage` | string | `stage: stage` |

Rules:
- Params are additive (all active params combine into one `where` object).
- Invalid or missing params are silently ignored (no 400 errors).
- The unfiltered lead set (no params) is still used for the summary bar and week-over-week table regardless of active filters.

### Pipeline summary bar

Displayed above the leads list, always shown (not affected by filters).

Computed server-side from the full unfiltered lead set:
- One entry per stage in `PIPELINE_STAGES` order
- Each entry: `{ stage, count, totalValue }` where `totalValue = sum(estimatedValue ?? 0)`
- Rendered as a row of stage chips with count badge and `$X` value (or blank if zero)
- Reuses existing `STAGE_COLORS` map

### Week-over-week table

Displayed between the summary bar and the leads list. No extra DB query — computed from the same full lead result.

| Column | This week | Last week |
|---|---|---|
| New leads | `createdAt >= now - 7d` | `createdAt` between `now-14d` and `now-7d` |
| Avg score | mean of `score` for leads with `scoredAt IS NOT NULL` | same window |

Rendered as a simple two-row HTML table. No charting library.

### Filter UI

A `<form method="GET" action="/leads">` above the leads list:
- Score range: two number inputs (min/max), current values reflected from `searchParams`
- Stage dropdown: `<select>` with all pipeline stages + "All stages" option
- Submit button: "Filter"
- Clear link: `/leads` with no params

No JavaScript required — plain form GET submission.

## Error handling

- Sales classifier: if messages array is empty, return `{ isSalesLead: false, ... }` — no error thrown.
- Filter params: `parseInt(minScore)` → `isNaN` check → ignore if invalid. Stage param: pass through to Prisma as-is; Prisma will return zero results for an invalid stage value (no enum in schema).
- `SalesPanel` "Generate closing draft" button: same error handling pattern as SupportPanel (shows inline error message on failure, no page reload).
