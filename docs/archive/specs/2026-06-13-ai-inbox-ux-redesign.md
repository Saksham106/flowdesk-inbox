## Problem Statement

FlowDesk currently feels like Gmail with AI cards bolted on. The layout is too narrow on desktop, the Home page is a stats widget rather than a command center, email snippets can show raw HTML, the conversation view is a pile of separate cards, and personal accounts see business/sales language. Users should feel like they have no reason to open Gmail â€” FlowDesk tells them what matters, hides noise, explains threads, drafts replies, and makes the next action obvious.

---

## Chosen Direction: Left Rail + Split-Pane Shell (Option B)

A persistent icon rail on the far left, a scrollable email list column in the middle, and the main content pane (Home or opened conversation) on the right. Routes remain stable (`/inbox` for list, `/conversations/[id]` for threads). Desktop-first MVP; mobile falls back to the existing full-page patterns.

## 3. Email List Column

The list column is always visible on desktop in the shell, filtered by whatever `?status` / `?q` param is active on the current route.

### Conversation row design

Each row contains:
- **Contact name** â€” bold if `needs_reply`, normal weight otherwise
- **Relative timestamp** â€” top right (2m, 1h, 3d)
- **Clean text snippet** â€” strip HTML from `lastMessage.body` before display; use a short helper that strips tags and truncates to ~80 chars. Never show raw HTML/CSS.
- **Status indicator** â€” colored dot + label: `â—Ź Needs Reply` (red), `â—Ź In Progress` (amber), `â—Ź Closed` (green), `No reply needed` (muted slate)
- **FYI/automated rows** â€” entire row is visually muted (50% opacity or slate text) to signal "safely handled"

### Filter tabs (inside list column header)

Compact pills above the list: `All (N)` Â· `Reply (N)` Â· `Progress` Â· `Closed`. Active pill is dark fill. Business accounts also get a `Sales` pill.

### Search

Search input inside the list header (above the filter pills). Submitting updates `?q=` on the URL. Matches existing search behavior.

### Snippet sanitization

Add a `stripHtmlToText(body: string, maxLength = 80): string` helper in `lib/email-body.ts`:
- If body starts with `<`, strip all HTML tags, collapse whitespace, truncate
- Otherwise, strip markdown (`**`, `_`) and truncate
- Never return raw `<` / `>` / CSS

## 4. Conversation View

The conversation page (`/conversations/[id]`) renders inside the shell. Main pane layout: `[thread+composer area | 240px sidebar]`.

### Thread area

- **Header** (sticky): contact name, email address, thread subject if different, status badge, "Mark closed / Mark needs reply" quick action button.
- **Message list**: vertically stacked `<article>` elements, top-to-bottom chronological order. Each message: sender avatar initial, sender name + Sent/Received tag, timestamp, body rendered via `EmailBody`. Inbound messages: white background. Outbound: light blue tint.
- **Reply composer** (inline below last message, NOT a separate card):
  - Title: "Reply to [Contact Name]"
  - If AI draft exists: `draft-badge` + pre-filled textarea. Primary CTA: "Approve & Send". Secondary: "Edit", "Suggest again", "Clear".
  - If no draft: `userInstruction` textarea + "Suggest reply" button. Textarea grows after suggestion appears.
  - "Or send directly" is a ghost-level secondary path, not a prominent section.
  - Risk warning (amber) appears inline above CTA if `riskLevel === "high"` or `escalationReason` is set.
  - Separator line between thread and composer is a `border-t-2` (heavier than message dividers).

### Right sidebar (240px)

Rendered only on desktop. Cards in priority order:

1. **Contact** â€” name + email. Always visible, compact. For non-email channels: SaveContactForm.
2. **Why this matters** â€” replaces the current "Assistant context" label with "Why this matters". Shows: State, Reason, Next step. "Suggest reply" button (primary). "Explain this thread" button (secondary/ghost). No `Handle this` label â€” use "Suggest reply" for personal, "Handle this" for business.
3. *(Business only)* **Support signals** â€” only if `isSupport && !isPersonal`
4. *(Business only)* **Sales pipeline** â€” only if `isSalesLead && !isPersonal`
5. *(Business only)* **Calendar hold** â€” only if `!isPersonal`
6. **Relationship** â€” `CollapsibleCard`, closed by default. Contains `personMemory` content.
7. **Work items** â€” `CollapsibleCard`, closed by default. Contains `stateRecord`, `inboxTasks`, `lead` (lead only shown if `!isPersonal`).
8. **No-reply card** â€” if `isAutoEmailConversation`: replace the entire sidebar with a single compact card: "No reply needed" + reason sentence. No action buttons.

### Personal account sidebar differences

- No `LabelSelect`
- No business profile warning in `AIDraftPanel` â€” personal accounts show "AI suggestions are temporarily unavailable" only if `canSuggest === false`, not a "add business profile" prompt
- "Suggest reply" instead of "Handle this" in the action button text
- No Lead section in Work items
- No SupportPanel, SalesPanel, CalendarHoldPanel

## 7. Visual Polish

### Spacing and widths

- App shell: `min-h-screen flex` â€” no max-width wrapper at shell level
- Thread reading column: `max-w-[640px]` within the thread area (for comfortable line length), centered with auto margins
- Sidebar: `w-60` (240px) on desktop, hidden on mobile
- List column: `w-[280px]` on desktop, hidden on mobile when conversation is open
- Gutters: `px-5 py-4` inside cards, `gap-3` between sidebar cards

### Text overflow

- All card titles: `truncate` or `break-words [overflow-wrap:anywhere]` depending on context
- Sidebar contact email: `break-all text-xs`
- Email list snippet: hard truncated at 80 chars by helper before render

### Empty states

- Empty list: "No conversations match your search." / "No conversations yet. Connect Gmail in Settings."
- Empty home (all caught up): single centered card "You're all caught up â€” nothing needs attention right now."
