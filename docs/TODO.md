The core loop only acts on *new* mail today, so a new user connects Gmail and sees an unchanged inbox. This phase closes that gap and makes the loop fast. See `docs/flowdesk-vs-reference-gap-analysis.md` → "Top MVP gap" for the Inbox Zero comparison.

- [x] **Retroactive first-pass on connect (flagship)** — shipped: `runOnboardingFirstPass` (`lib/agent/onboarding-first-pass.ts`) labels a bounded batch (30 days / 40 threads) of the user's existing inbox via the same `reconcileGmailLabelsForChannel` path the "Fix Gmail labels" button uses; classification is deterministic (zero LLM cost). The OAuth callback redirects a *new* connection (not a reconnect) to `/onboarding`.
- [x] **Proof screen** — shipped: `/onboarding` (`app/onboarding/`) runs the first-pass and renders "N emails organized" with a per-label breakdown and sample threads, aggregated from the `gmail.labels.queued` audit rows the pass produced, before a "Go to your control room →" CTA. Live-verified end-to-end (4 seeded threads → correct labels + breakdown).
- [x] **Performance + correctness pass on hot paths** (from the 2026-07-08 perf audit) — shipped: home page collapsed ~6 serialized DB round-trips into one `Promise.all` (biggest latency win); fixed the latest-message correctness bug (home fetched the *oldest* `HOME_MESSAGE_LIMIT` messages then picked newest among them — wrong for threads longer than the limit; now `orderBy: desc`); added `@@index([tenantId, status, lastMessageAt])`; guarded the write-on-read (`isRead: false`) that rewrote every message row on each thread open; merged the conversation page's second query group into the first. **Deferred (with reason):** caching the command-center fetch in `unstable_cache` — its query returns nested `Date` objects that JSON-cache serialization would stringify, breaking `latestMessage`/date math; needs date rehydration first. Routing the inbox `q` search through the existing `Message.searchVector` GIN index instead of ILIKE — off the default hot path.
- [x] **Tighten the default path** — shipped: deleted the dead `/digest` route (bare `redirect("/inbox")`) plus its two orphaned components, removed Digest from the mobile nav (`lib/app-navigation.ts`), and repointed the Settings Follow-Up copy's "daily digest" link at the control room. Booking failures never strand the session: the error is recorded, audited, surfaced in the Scheduling panel, and retryable there.

- Improved deterministic email classification (reported bug: marketing emails labeled `Newsletter`). Root cause was ordering: the classifier matched the generic unsubscribe/footer signal → `newsletter` *before* the marketing rules, so every sale/offer email (which legally must carry an unsubscribe link) was mislabeled a newsletter; and email-service-provider domains (Mailchimp, Klaviyo, Constant Contact, HubSpot, Marketo, etc.) were listed as notification senders, forcing their newsletters/marketing to `notification`. Reworked to **content-first** ordering — promotional intent → `marketing`, editorial/subscribed content → `newsletter`, then sender/pattern notification rules, then a generic bulk-list fallback → `newsletter` — with broader high-precision promo vocabulary, positive editorial newsletter signals, split marketing/newsletter sender-subdomain hints, and the pure-ESP domains dropped from the notification list. Marketing/newsletter definitions follow Inbox Zero's. 8 new tests; verified across a 15-email real-world sweep (all labels correct, including a GitHub notification carrying an unsubscribe footer staying `notification`, and a personal "garage sale" email staying `needs_reply`).
- Onboarding retroactive first-pass + proof screen (the MVP flagship — closes the "new user connects Gmail and sees an unchanged inbox" gap identified from Inbox Zero's onboarding). On a fresh Gmail connect the OAuth callback redirects to `/onboarding`, which calls `runOnboardingFirstPass` (`lib/agent/onboarding-first-pass.ts`): it labels a bounded batch (30 days / 40 threads) of the user's *existing* inbox through the same `reconcileGmailLabelsForChannel` path the "Fix Gmail labels" button uses, then renders "N emails organized" with a per-label breakdown and sample threads — aggregated from the `gmail.labels.queued` audit rows the pass produces — before a "Go to your control room" CTA. Classification is deterministic, so the backlog pass costs zero LLM spend (FlowDesk can go deeper than Inbox Zero's per-message-LLM 20-message pass). A reconnect (credential refresh on a known account) skips onboarding and returns to Settings. Live-verified end-to-end: 4 seeded existing threads → correct labels (newsletter→Read Later+Newsletter, marketing→Handled+Marketing, notification→Handled+Notification, question→Needs Reply), accurate breakdown, no console errors.
- In-process background job scheduler (`lib/scheduler/`): the biggest single gap toward a working MVP. All 13 `/api/cron/*` routes existed but nothing in the deployment (Railway, single `next start` process, no external cron service configured) ever called them — the classification pipeline, Gmail writeback retries, label/state reconciliation, follow-ups, snooze resurfacing, workflow runs, lead sequences, and Outlook sync were all silently dead in production; inline draining (added earlier) covered the common case but not retries or drift correction. Extracted each route's inline business logic into an importable `lib/` function (7 routes had logic inline; 6 already delegated to one), then added a registry-driven scheduler that boots once via Next.js's `instrumentation.ts` hook and calls each function directly on its own interval (1 min for classification/writeback, 5–30 min for retries/drift/workflows, 6–24h for maintenance) — no HTTP round-trip, no `CRON_SECRET` dependency, so a misconfigured secret can no longer silently block everything. Overlap-guarded per job (a slow run skips its next tick rather than double-processing) and failure-isolated (one job throwing never stops another or crashes the interval). Status tracked via a `globalThis`-backed singleton (Next.js can instantiate a module more than once per process — the same reason `lib/prisma.ts` uses `global.prisma` — so a plain module-level variable silently failed to share state between `instrumentation.ts` and the route that reads it; caught live during verification, not by the test suite, and fixed the same way) and exposed at `GET /api/admin/scheduler-status` (`CRON_SECRET`-gated). The `/api/cron/*` HTTP routes are unchanged and still work for manual triggering. Verified live against a `next start` production build (not just `next dev`): the two `runOnStart` jobs (agent-jobs, gmail-writeback) ran automatically and successfully against the real database within seconds of boot, with no external caller involved.
- Fixed a real "labels are wrong, not just missing" bug found via live user testing: `projectFlowDeskLabelsForConversation` only read `attentionCategory`/`emailType` from the *persisted* `ConversationState`, which requires the AI classification job to have actually run. For a conversation it never ran for (the common case on an account that predates the classification pipeline, or where the job never executed), `deriveWorkflowStatus` fell through to a uniform "needs_reply" default — so newsletters, notifications, and already-handled mail all got labeled "Needs Reply" in Gmail even though the FlowDesk Inbox app itself showed them correctly (the app's dashboard uses richer, independently-computed signals the label path didn't have). Added a deterministic (no AI, no DB) `classifyEmailType` fallback for never-classified conversations. Verified live: a seeded newsletter with no `ConversationState` row now correctly queues "Read Later" instead of "Needs Reply". Also made the "Fix Gmail labels" button diagnose *why* nothing changed instead of a flat "already up to date" — it now surfaces the tenant's automation level and calls out explicitly when that's the actual blocker (queued=0 was previously indistinguishable between "genuinely nothing to fix" and "automation level silently blocked everything").
