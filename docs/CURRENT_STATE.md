# Current State

Last updated: 2026-07-11 (Gmail-first feedback and writing intelligence — Gmail label additions/removals become durable corrections, clear-label overrides survive reconciliation until a genuinely new inbound email, deterministic/AI classification uses bounded evidence, native Gmail drafts stay singular and withdraw after a manual Gmail reply, reply training excludes forwarded/automated/list/duplicate samples, and Training includes enforceable writing preferences including a no-em-dashes rule.)

FlowDesk is a Gmail-native AI email operator for individuals and small businesses. Gmail is the primary daily workspace; the FlowDesk web app is the agent control room for setup, preferences, approvals, audit history, training, and power-user review.

## Implemented

### Platform

- NextAuth credentials authentication with tenant-scoped data isolation.
- **B2C, one universal experience.** There is no personal/business account identity in storage. An opt-in **Sales & CRM mode** capability (`Tenant.salesCrmEnabled`, off by default) enables lead scoring, sales/support signal classification, revenue-at-risk, reports, meeting prep, and business-tone AI framing; when off (the default baseline) all of that is suppressed at the prompt and sync layers. The capability is the source of truth — the legacy internal `personal`/`business` mode threaded through prompts/sync is derived from it via `accountModeFor` (`lib/tenant-capabilities.ts`), and it toggles in Settings → Features. The deprecated `Tenant.accountType` column and `AccountType` enum have been removed.
- **Tight default path (MVP):** with Sales & CRM mode off, half-built peripheral surfaces are not reachable from the default navigation. The dead `/digest` route (a bare `redirect("/inbox")`) was deleted along with its unused components and its mobile-nav entry; the Follow-Up Automation copy in Settings now points at the control room instead. Deferred surfaces are gated behind the capability rather than deleted: `/risk-radar` redirects to `/inbox` like the rest of the sales cluster; the conversation right rail hides the Second Brain facts panel and the Scheduling panel; Settings hides the Workflows and Snippets & Playbooks panels, and shows the Outlook connect entry only when the capability is on **or** a channel already exists (existing connections stay manageable and disconnectable). Feature code is untouched — only entry points were removed or gated. (The Google Drive connect entry, formerly gated the same way, was removed entirely — see Integrations below.)
- Audit logs, approval requests, automation run traces, and undo for selected reversible actions.
- Unified approvals: `ApprovalRequest` is the single approval primitive (`step: "send"`, source in `metadataJson`). Every draft that reaches `proposed` (suggest, manual edit, meeting follow-up) gets one pending request; approve/send/clear/autopilot-send resolve it (`lib/agent/approvals.ts`), and `/approvals` decisions project back onto `Draft.status` — approve marks the draft approved, reject clears it and withdraws any Gmail-native draft. Cleared drafts cancel their request (`cancelled` status).
- **Background job scheduler** (`lib/scheduler/`): every recurring pipeline (AI classification, Gmail writeback retries, label/state reconciliation, follow-ups, snooze resurfacing, workflow runs, lead sequences, Outlook sync, snippet mining, weekly value snapshots, data retention — 14 jobs total) runs in-process rather than depending on an external caller hitting the `/api/cron/*` routes. Next.js's `instrumentation.ts` `register()` hook (needs `experimental.instrumentationHook: true` in `next.config.js` on Next 14; unflagged in 15+) fires once when the server process starts and calls `startScheduler()`, which registers a `setInterval` per job pulled from `buildJobRegistry()` (`lib/scheduler/jobs.ts`) — intervals range from 1 minute (classification, writeback) to 24 hours (snippet mining, value snapshots, data retention), and jobs call their underlying `lib/` function directly rather than making an HTTP request to their own route, so there's no `CRON_SECRET` dependency for the scheduler to function. Each job run is overlap-guarded (a slow run's next tick is skipped, not queued) and failure-isolated (one job throwing is caught, logged, and never stops the interval or any other job). Status (last run time, duration, result, error, run/error counts) is tracked in a `globalThis`-backed singleton — plain module-level state doesn't reliably survive across the separate module instances Next.js can create for `instrumentation.ts` vs. a route handler within the same process, the same reason `lib/prisma.ts` uses `global.prisma` — and exposed at `GET /api/admin/scheduler-status` (`CRON_SECRET`-gated, since job status spans tenants). Set `SCHEDULER_ENABLED=0` to disable (e.g. if an external scheduler is introduced later). The `/api/cron/*` HTTP routes are unchanged and still work for manual/on-demand triggering; they were refactored to call the same shared `lib/` functions the scheduler uses, so there is exactly one implementation of each job's logic.
- **Data retention** (`lib/agent/data-retention.ts`, `data-retention` job, daily + on boot): prunes unbounded operational tables to a rolling window — `AuditLog` 30 days, `AiUsageEvent` 90 days (must exceed a full calendar month because `lib/ai/budget.ts` aggregates the current month for spend limits), `GmailPushEvent` and `OutlookSyncEvent` 30 days. `EmailWritebackQueue` is deliberately NOT pruned: it is upserted on `@@unique([conversationId, action])` (bounded by conversation count, not mail volume) and its completed/acknowledged rows feed the echo-suppression check in `queueFlowDeskLabelWriteback`. These per-action receipt tables are what filled the Postgres volume in the 2026-07-12 outage; with retention they plateau instead of growing forever. Product data (messages, conversations) is never touched. Steady-state write volume is also gated at the source (2026-07-13): no-op receipts are no longer written at all — no AiUsageEvent for skipped LLM calls (`person_memory.cache_hit` / `too_few_messages` / `policy_skipped`), no `conversation_state.synced` or `person_memory.synced` audit row unless the row actually changed (`deterministicStateUnchanged` in `lib/agent/work-item-sync.ts` strips classifier-owned metadata keys before comparing, since the email classifier re-merges those every sync), and Outlook delta no longer rewrites a stored message's full row (body included) on read/category touches — only `isRead`. Full storage model, capacity math, and do-not-prune warnings: `docs/storage-and-capacity.md`.

### Automation trust ladder

A per-tenant automation level (`AutopilotSetting.automationLevel`, 0–5) is the user-facing ceiling on what the agent may do on its own. It is mapped onto the existing gates (`lib/agent/automation-level.ts`) and never loosens them — an automated action runs only when the level allows it AND every pre-existing confidence/policy/budget/cap gate passes. User-initiated actions are never level-gated.

| Level | Meaning | Gmail labels | Gmail drafts | Auto mark-read/archive | Auto-send |
|-------|---------|--------------|--------------|------------------------|-----------|
| 0 | Read-only insights | no | no | no | no |
| 1 | Suggest in dashboard only | no | no | no | no |
| 2 | Apply labels in Gmail (new-tenant default) | yes | no | no | no |
| 3 | Create Gmail drafts | yes | yes | no | no |
| 4 | Mark low-risk read / archive safe categories | yes | yes | yes | no |
| 5 | Auto-send approved categories | yes | yes | yes | yes* |

*Auto-send requires Level 5 **and** autopilot enabled **and** all existing gates (learned profile, risk, confidence + per-category thresholds, allow-list, budget, daily cap, failure auto-disable). Levels ≤ 4 can never auto-send; the gate fails closed on a missing level. Auto-booking a calendar event from a detected scheduling confirmation (`auto_book_event`) also requires Level 5 — an invite emails the counterparty, the same outward-facing tier as auto-send; below Level 5 the confirmation raises a `book_event` ApprovalRequest instead, and user-initiated booking is never level-gated. No automatic mark-read/archive path exists yet — the Level 4 gate is defined for future callers. Level changes require an explicit confirm in settings, are audited (`automation_level.changed`), and take effect immediately. Existing tenants were migrated without increasing autonomy: autopilot-enabled → 5, everyone else → 3 (labels + Gmail drafts were already unconditional); tenants without a settings row derive Level 3.

### Gmail

- OAuth connect/callback with encrypted token storage. Access token is auto-refreshed by the googleapis client; the `"tokens"` event handler persists refreshed tokens back to the DB.
- Expired or revoked credentials (`invalid_grant`) set `lastSyncStatus: "needs_reauth"`, surface a "Reconnect Gmail" CTA in both the inbox control and settings, and stop all automatic polling until reconnected.
- Incremental history sync via the Gmail History API with `labelId: INBOX` filter. Falls back to a recent-message scan when the history cursor is stale (cursor resets are timestamped in `lastHistoryFallbackAt` for UI visibility).
- Gmail FlowDesk label additions/removals observed through History API are treated as Gmail-first user corrections, not drift. A label removal records an explicit clear-label override that blocks automatic re-projection; it is released only when a genuinely new inbound message arrives or the user makes an explicit FlowDesk change. Own writeback echoes are consumed once, so later matching manual Gmail edits are still learned.
- Pub/Sub push notifications via renewable watches. Each notification is persisted in `GmailPushEvent` with idempotency on the Pub/Sub message ID; duplicate deliveries are no-ops.
- Per-channel sync lock via `updateMany` on `syncLockExpiresAt`. Parallel sync attempts skip and return 202 without duplicating work. Lock owner is implicit (row-level); lock expires after 2 minutes.
- Local read/archive/trash/unsubscribe writeback to Gmail. Failed mark-read writes queue into `EmailWritebackQueue` (renamed from `GmailWritebackQueue` — the queue is now shared with Outlook, see Outlook section below) and are retried by cron with exponential backoff.
- Gmail-native label projection runs on manual workflow/status changes and automatically after classification and draft creation (`projectFlowDeskLabelsForConversation` in `lib/agent/work-item-sync.ts`, skipped when the user has manually overridden state or the tenant's automation level is below 2; the projection derives the workflow status from `conversation.userState` first, so a manual Waiting On / Read Later / Done choice is never reverted by re-projection). `lib/gmail-labels.ts` maps FlowDesk state to flat, top-level user-facing Gmail labels — 6 workflow-state labels (`Needs Reply`, `Needs Action`, `Waiting On`, `Read Later`, `Handled`, `Autodrafted`) plus 4 content-type labels inspired by Inbox Zero's taxonomy (`Newsletter`, `Marketing`, `Notification`, `Calendar`), driven by the deterministic classifier's `emailType` — no `FlowDesk/` namespace prefix. Retired: `Follow Up` (overdue tracking is app-only now, see `followUpDueAt`/`WaitingOnSection`), `Important` (the underlying Lead/Pricing/Complaint signal is in-app only), `Low Priority` (superseded by the specific content labels). Queues `apply_labels` jobs in `EmailWritebackQueue` and records `gmail.labels.queued` audit entries. **Echo-loop guard:** queueing is idempotent — if the conversation's last `apply_labels` job asked for the identical label set, `queueFlowDeskLabelWriteback` returns without re-queueing (pending/processing rows are returned untouched so retry backoff survives; completed/acknowledged rows suppress the re-queue for 24h, after which reconcile can converge genuine hand-edited drift). Without this, FlowDesk's own writeback echoed back through Gmail push → sync → re-projection and re-armed the queue forever (the 2026-07-12 storage/API runaway: ~180k redundant label queues per day against ~110 conversations). Accounts connected before the labels were flattened may still have legacy nested `FlowDesk/*` labels; `reconcileLegacyFlowDeskLabels` (`lib/google.ts`) renames legacy-only labels in place, removes duplicate legacy labels when the flat label already exists, and deletes the now-empty `FlowDesk` parent whenever the ensure/apply paths touch Gmail. Persisted `GmailLabelMapping.canonical` values were flattened by migration. Each label gets a background/text color (`FLOWDESK_GMAIL_LABEL_COLORS`, values copied from Inbox Zero's verified-working palette — Gmail's color API only accepts a fixed, unenumerated-in-types set of pairs) applied via a **separate best-effort call** after the label is created/renamed (`applyLabelColorBestEffort`), so a rejected color pair can never block the label from existing or being applied to a thread — a prior version bundled color inline with create/rename/patch, which meant one bad color pair could abort labeling for every thread.
- The app's own inbox list (`app/components/AppListColumn.tsx` desktop, `app/mail/page.tsx` mobile — the list lives on the dedicated `/mail` route, not the home screen) mirrors the Gmail content-type taxonomy: Newsletter/Marketing/Notification/Calendar filter pills (shared matching logic in `lib/content-type-filters.ts`, `?type=` query param) sit alongside the existing status pills, and each conversation row shows a small content-type badge (`ContentTypeBadge` in `app/components/badges.tsx`) next to its workflow-status badge when `emailType` is one of the four content types. `QuietlyHandledBreakdown` (`lib/agent/command-center.ts`) also tracks a `calendar` bucket (previously silently folded into `other`), and its Home-view banner pills (`QuietlyHandledBanner.tsx`) link directly to the matching `?type=` filter instead of being static counts.
- **Onboarding wizard:** new signups land on `/onboarding` (`app/onboarding/OnboardingWizard.tsx`, via the signup branch's post-signin `callbackUrl`), a two-step skippable wizard. Step 1 — Connect Gmail or Outlook (`GET /api/connectors/{gmail,outlook}/connect`; Outlook is offered whenever `MICROSOFT_CLIENT_ID/SECRET` are configured). Picking a provider first shows a pre-OAuth instructions interstitial (both consent screens show an "unverified app" warning while Google/Microsoft verification is in review; the interstitial walks through Advanced → Continue and explains that OAuth means FlowDesk never sees the password and data is deleted on disconnect). A fresh connection (not a reconnect) redirects from the OAuth callback back to `/onboarding?connected=<email|outlook>`, where the user picks how many recent emails to organize (0 / 25 / 50, default 25; 0 skips the pass), which calls `runOnboardingFirstPass` (`lib/agent/onboarding-first-pass.ts`) via a session-authed `POST /api/connectors/gmail/first-pass` with `{limit}`. It labels a bounded batch (30 days / chosen thread count) of the user's **existing** inbox through the shared `reconcileGmailLabelsForChannel` path, so the real inbox is organized in the first session instead of only new mail being labeled as it arrives, then renders "N emails organized" with a per-label breakdown and sample threads (aggregated from the `gmail.labels.queued` audit rows the pass produces). Classification is deterministic, so the backlog pass costs no LLM spend. Gated on automation level ≥ 2 (the new-tenant default); a lower level surfaces an honest "raise your level" message instead of a silent no-op. Step 2 — Train your style (`POST /api/personal-profile/train`) with a "what we learned" tone/greetings/sign-offs summary, then a "Go to your control room" CTA. Steps derive from real state (`Channel` type `email`, `LearnedReplyProfile`) via `resolveOnboardingStep` (`lib/onboarding.ts`); fully-set-up visitors are redirected to `/home`, and every step is skippable.
- FlowDesk labels are bootstrapped in Gmail on OAuth connect and manual sync (`ensureFlowDeskLabels`); per-label enable/disable and rename settings persist in `GmailLabelMapping` via `/api/gmail-label-settings` and the settings page. Label-set-ensure + re-projection logic is shared (`lib/agent/gmail-label-reconcile.ts`, `reconcileGmailLabelsForChannel`) between two callers: `GET /api/cron/gmail-label-reconcile` (recurring maintenance, per-channel-batched at 14 days / 50 conversations, re-creating labels the user deleted and re-projecting drift) and `POST /api/connectors/gmail/relabel` (session-authenticated, triggered by the "Fix Gmail labels" button in Settings → Gmail behavior, a wider one-time catch-up at 365 days / 100 conversations for accounts whose labels were frozen from before a reliability/color fix — e.g. label writeback jobs that permanently failed under old code get reset to pending and retried, since `queueFlowDeskLabelWriteback`'s upsert resets `status`/`attempts` on re-queue (failed rows always re-queue; only identical recently-completed label sets are skipped by the echo-loop guard)). Both only enqueue through `EmailWritebackQueue`, so overlapping runs are safe. Full/manual Gmail sync imports a bounded recent batch only: default 25 inbox threads, explicit test cap accepted by the manual sync API, and hard-capped at 50 — and does **not** revisit already-synced conversations when run incrementally (the common case for a returning user), so it does not substitute for the relabel action above.
- Gmail-native drafts: proposed FlowDesk reply drafts are written to Gmail as real drafts through `create_draft` writeback jobs (`lib/gmail-drafts.ts`), at automation Level 3+ (below that the draft stays dashboard-only). Draft source metadata keeps one FlowDesk draft per thread, invalidates stale drafts after newer inbound mail, and queues withdrawal after a manual Gmail reply. A failed old-draft deletion retries rather than creating a duplicate; `withdraw_draft` jobs are never level-gated (cleanup keeps working after a tenant lowers their level).
- Reply-style training accepts only verified native Gmail Sent samples, excluding forwards, quoted material, automatic/list mail, and duplicates. The Training settings let users override learned style with greetings, sign-offs, formality, length, phrases to avoid, custom instruction, and an enforceable no-em-dashes preference.
- `GET /api/cron/gmail-writeback` processes `mark_read`, `apply_labels`, `create_draft`, and `withdraw_draft` jobs. Label jobs create missing Gmail labels, apply the current FlowDesk labels to the thread, and remove stale labels from the managed FlowDesk label set. Before mutating, `applyFlowDeskLabelsToGmailThread` reads the thread (`threads.get`, minimal) and drops add/remove ids that would not change any message's labels; if nothing would change it skips `threads.modify` entirely — a redundant modify advances mailbox history and re-enters the pipeline as a push echo. Every job is claimed atomically (pending → processing via `updateMany`) so overlapping cron runs never double-process; failures retry with exponential backoff (`nextWritebackAttemptDate`) and fail out permanently after `GMAIL_WRITEBACK_MAX_ATTEMPTS`. Every job resolution is linked to a human-readable audit entry: completions record `gmail.writeback.completed` (action, conversation, thread/labels/draft detail, or the skip reason for silent no-ops) and permanent failures record `gmail.writeback.failed` (error, attempt count) — transient retries are not resolutions and write no audit row. Permanently-failed jobs are not stuck forever: a successful OAuth reconnect requeues every failed writeback job on that channel (`requeueFailedWritebacksForChannel`, `lib/email/writeback-requeue.ts`, audited as `<provider>.writeback.requeued`) — with credential failures like `invalid_grant`, it was the credential, not the job, that was broken. An empty label set is a valid payload meaning "remove all FlowDesk labels" and is queued only for threads that were labeled before, so no-label transitions clean up stale labels without spamming untouched threads.
- Settings now shows **Gmail operator health** (`lib/gmail-operator-health.ts`, `app/settings/GmailOperatorHealthPanel.tsx`): a tenant-scoped summary of Gmail auth/sync, push watch/push failures, Gmail writeback backlog/failures, and pending/running/failed `AgentJob`s. This makes stalled label/draft/classification pipelines visible in the product rather than only in cron logs.
- **Waiting-on / follow-up lifecycle** (Phase C): outbound replies that plausibly expect a response (deterministic heuristic `outboundMessageExpectsReply` in `lib/agent/follow-up.ts` — no LLM, personal and business modes alike) move the conversation to Waiting On. FlowDesk sends transition directly in the send path; replies sent natively in Gmail are detected during history sync (`conversation.waiting_on_detected` audit). An inbound reply self-heals: back to Needs Reply, pending follow-up jobs cancelled, stale waiting-on attention cleared, labels re-projected (`conversation.waiting_on_cleared`). After a tenant-configurable delay (`FollowUpSetting.staleAfterDays`, business days, default 3) the follow-up cron's label sweep (`runFollowUpLabelSweep`) re-projects labels for overdue waiting-on conversations to catch drift and audits `follow_up.due_labeled`; there is no distinct `Follow Up` Gmail label — overdue tracking surfaces in-app only (`WaitingOnSection`) — and the sweep runs for every tenant, while `FollowUpSetting.enabled` continues to gate only the automated follow-up job pipeline. No auto-sent follow-ups — labels and dashboard surfacing only.
- Gmail cron routes reject bearer-token auth when `CRON_SECRET` is unset, preventing accidental `Authorization: Bearer undefined` access in misconfigured deployments.
- Local user intent (read state, conversation status, attention category) is stored separately from raw Gmail state (`gmailUnread`, `gmailLabelIds`, `gmailRawState`) so provider syncs cannot overwrite explicit user choices.
- `GmailStateReconcile` cron detects local-read/Gmail-unread drift; auto-reconciles non-user reads and queues writeback for user-initiated reads.

### Outlook

- OAuth connect/callback with encrypted token and cursor storage.
- Leased incremental delta sync via Microsoft Graph `/me/mailFolders('inbox')/messages/delta`. The encrypted `deltaLinkEncrypted` cursor is persisted after each page so bounded cron runs can continue large initial imports across invocations without restarting.
- Lease uses a random owner ID (`syncLeaseId`); the completion update requires the same ID so a stale worker cannot release a newer worker's lock. HTTP 410 / invalid-cursor clears the saved cursor and schedules a fresh start.
- Webhook endpoint queues authenticated `OutlookSyncEvent` hints only (constant-time client-state comparison; message content and tokens are never stored). Graph delta sync happens in cron, not inline with the webhook response.
- Bounded cron: at most 25 queued events, 25 fallback credentials, and 25 subscription renewals per run.
- Admin rekey covers Outlook access/refresh tokens plus encrypted delta links and subscription client state, so key rotation does not strand incremental sync metadata.
- **Outlook parity** (2026-07-12): Outlook now shares the same provider-neutral writeback pipeline as Gmail. `lib/outlook-mailbox.ts` projects FlowDesk workflow/content-type state onto Outlook categories (`ensureFlowDeskCategories`, `applyFlowDeskCategoriesToConversation`) and implements mark-read, archive/restore/trash, and create/delete draft-reply via Microsoft Graph. Writes queue through the shared `EmailWritebackQueue` (see Gmail section above — the table was renamed from `GmailWritebackQueue` since it now serves both providers) and are dispatched by provider in `lib/agent/email-writeback-processor.ts`, replacing the old Gmail-only cron. Outlook delta sync feeds category changes back into local state as user feedback (`lib/agent/outlook-category-feedback.ts`) the same way Gmail label edits do. The state/label reconcile crons (`email-state-reconcile`, `email-label-reconcile` in `lib/scheduler/jobs.ts`) and the `POST /api/connectors/outlook/relabel` catch-up route are provider-neutral, reusing the same reconcile logic as Gmail. Settings → Connect shows an **Outlook operator health** panel (`lib/outlook-operator-health.ts`, `summarizeOutlookOperatorHealth`) alongside the Gmail one, and the same "Fix labels" button (`FixGmailLabelsButton`, provider-parameterized) triggers the Outlook relabel route. One limitation vs. Gmail: because Outlook's delta feed is inbox-scoped it cannot observe outbound sends, so a reply the user sends manually from Outlook leaves the stale FlowDesk mailbox draft in place until the next `create_draft` job reconciles it (Gmail withdraws the draft the moment its sync sees the manual reply).

### Email rendering

- Gmail MIME tree walker collects `text/html` and `text/plain` parts at up to 12 levels of nesting. HTML is preferred for rendering; plain text is used for AI prompts and snippet generation.
- HTML is sanitized with a strict allow-list and rendered in a sandboxed iframe. Remote images and other external network fetches are blocked by default. Users can explicitly load HTTPS images per message; the choice is not persisted (privacy-preserving by design).
- A forced light-mode color scheme prevents dark-mode email templates from rendering black in the iframe.
- Plain-text bodies are auto-linked. Gmail `cid:` inline images are resolved from MIME attachments into size-capped safe `data:` URIs at sync time (`resolveInlineCids` in `lib/google.ts`); messages synced before that fix keep unresolved `cid:` references.

### Inbox intelligence

- **Command center** (`lib/agent/command-center.ts`): pure analysis module — accepts pre-fetched plain objects, never calls Prisma directly. Pages own data fetching. This boundary keeps the analyzer independently unit-testable and lets page data shapes evolve without touching the scoring logic.
- **Email classifier** (`lib/agent/email-classifier.ts`): fully deterministic, no DB or AI calls. After the transactional/action rules (OTP, login approval, password/verify, security, billing, delivery, calendar), the content-type rules run **content-first**: promotional/sales intent → `marketing`, then editorial/subscribed content → `newsletter`, then known notification senders/patterns → `notification`, then no-reply local part, then weak sender-subdomain hints, then a generic bulk-list (unsubscribe/footer) fallback → `newsletter`. This ordering matters: marketing and newsletter definitions follow Inbox Zero's (marketing = "promotional emails about products, services, sales, or offers"; newsletter = "regular content from publications, blogs, or services I've subscribed to"). Checking the unsubscribe-footer signal first — as an earlier version did — mislabeled every sale/offer email as a newsletter, and listing email-service-provider domains (Mailchimp, Klaviyo, etc.) as notification senders forced their newsletters/marketing to `notification`; both are fixed by checking content before sender/footer and dropping the ESP domains from the notification list. Result stored in `ConversationState.metadataJson.emailType`; no schema migration needed.
- **Sales & CRM capability gating**: with the capability off (the B2C default), the classify prompt uses the baseline personal framing with no sales/lead/business language, and lead scoring (`scoreLeadForConversation`) and sales signal classification are skipped entirely. Turning on Sales & CRM mode enables all of them. The gate reads `Tenant.salesCrmEnabled` at each boundary and derives the internal mode via `accountModeFor`.
- **Reply-style learning**: reads FlowDesk outbound DB rows first; falls back to `fetchGmailSentSamples` (Gmail SENT label) when DB sample count < 5. Source counts stored in `sourceStatsJson` for transparency in the settings UI.
- **User-facing workflow status** (`Conversation.userState`): five clean states — Needs Reply, Draft Ready, Waiting On, Read Later, Done. `userState` is the canonical user choice; when null, `deriveWorkflowStatus()` computes it from DB status + AI attention category + draft state. AI signals (`attentionCategory`, `emailType`) remain in `ConversationState` as secondary read-only context, not primary user controls.
- **Reply workflow transitions**: AI draft generation/editing resets the conversation to Draft Ready inputs (`status: needs_reply`, `userState: null`) so proposed drafts win over stale Done/Waiting state. Successful sends create the outbound message, persist the conversation as Waiting On by default, clear the composer and saved draft text, and expose immediate Done / Waiting On next-step actions in the conversation composer. The composer supports CC/BCC behind a Gmail-style collapsed Cc/Bcc toggle: recipients are validated server-side (`lib/conversations/recipients.ts` — format, header-injection rejection, 25-per-field cap), sent as `Cc:`/`Bcc:` headers in the raw Gmail MIME (Gmail delivers to Bcc recipients and strips the header from other recipients' copies) or Graph `ccRecipients`/`bccRecipients` for Outlook, persisted on the outbound `Message` row (`ccEmails`/`bccEmails`), and recorded in the `conversation.send` audit payload.
- Manual corrections via `WorkflowStatusSelect` in the conversation right rail; learned sender/domain rules. Explicit user corrections always take precedence over learned rules and AI classification.
- Tasks, leads, follow-ups, risk radar, meeting prep/follow-up, weekly value reports, and revenue-at-risk reporting.
- AI drafts with knowledge-document citations, learned reply style, tenant-wide daily/monthly budget limits, per-feature metered AI usage, and human approval gates. Inbox chat and the agent-rule compiler both enforce budget checks and resolve an OpenRouter API key (`lib/ai/openrouter-keys.ts`) at request/runtime boundaries, so missing keys return controlled errors instead of import-time crashes. Budget-rejection errors point users at Settings → AI usage.
- **OpenRouter as the default AI provider** (`lib/ai/openrouter-keys.ts`): one OpenRouter child key is provisioned per FlowDesk user (not per tenant, not shared) via `OPENROUTER_MANAGEMENT_API_KEY`, so per-user usage and spend are individually capped and tracked; outside production only, a single shared `OPENROUTER_API_KEY` fallback is allowed when no management key is configured. `OPENAI_API_KEY` is no longer required for app AI. Every `AiUsageEvent` records `provider`, `providerKeyHash`, `providerGenerationId`, and both `estimatedCostUsd` and the provider-reported `actualCostUsd`.
- **Settings → Data → AI Usage** (`app/settings/AiUsagePanel.tsx`, `lib/ai/usage-summary.ts`): read-only per-feature breakdown of AI spend (inbox chat, rule compilation, classification, drafting, lead scoring, reply-style learning, person memory) for the current UTC day and month, plus remaining daily/monthly budget and blocked-call counts, alongside the existing AI Spend Budget limit editor. `summarizeAiUsage` is a pure, unit-tested helper — the settings page fetches the month's `AiUsageEvent` rows (including `actualCostUsd`, `userId`, `provider`) and passes them in (command-center pattern); costs prefer the provider-reported `actualCostUsd` over the pre-call `estimatedCostUsd` when present.
- Search, budget-metered inbox chat, person memory, attachment extraction, phishing warnings, VIPs, snooze, and Clean Inbox bulk actions.
- **Clean Inbox — sender-grouped cleanup, split into a Bulk Archive / Bulk Unsubscribe / Analytics tab strip** (`app/clean-inbox/{page,unsubscribe,analytics}.tsx`, `lib/agent/sender-cleanup.ts`): cleanable mail (newsletters/marketing + quietly-handled/FYI) is grouped by sender email, biggest-first, each group showing a count and sample subjects. Bulk Archive (`/clean-inbox`) offers **Archive all** / **Unsubscribe + archive** on every group; Bulk Unsubscribe (`/clean-inbox/unsubscribe`) is the same grouping filtered to senders with a working unsubscribe link; Analytics (`/clean-inbox/analytics`) shows counts by content type and top sender domains, computed in-memory over the same query (no new schema). `groupCleanupBySender` is a pure, unit-tested helper that bakes in the safety skip rules — a conversation is never offered for bulk cleanup if it needs a reply, is being waited on, is flagged important, or is a receipt/invoice. The batch archive/unsubscribe routes now actually **archive in the provider mailbox** (Gmail: remove the `INBOX` label via `archiveGmailThread`; Outlook: move the conversation out of the Inbox folder via `archiveOutlookConversation`, routed through the shared writeback adapter — best-effort and per-thread isolated in `lib/clean-inbox-email.ts`) so cleanup leaves the real inbox, not just the FlowDesk row; the 1-hour undo restores it (`INBOX` re-added for Gmail, moved back to Inbox for Outlook). Default is archive (not trash); everything is reversible for one hour. Outlook wrinkle: archiving moves the conversation out of the inbox-scoped delta feed's view, so it reports the thread `@removed` on the next sync — harmless, since the conversation is already closed locally and undo re-syncs it.

### Work items and classification persistence

- `lib/agent/work-items.ts` extracts task and lead candidates from conversation analysis. `lib/agent/work-item-sync.ts` persists them with tenant-scoped upserts and audit logs.
- Conversation-state persistence never overwrites records with source `"user_override"`; only deterministic-source state records are updated by sync. `InboxTask` field edits are protected the same way: editing a task's due date or status records the field in `metadataJson.userEditedFields` and flips `source` to `"user"` (`lib/agent/user-edited-fields.ts`), and the work-item sync update branch skips user-owned fields while still refreshing unedited ones.
- Fire-and-forget work-item sync calls catch and log async rejections, preventing background extraction failures from becoming unhandled promise rejections.
- `ConversationState.metadataJson` is a free-form JSON blob — all classification metadata (attention category, reason, confidence, email type, action code, expiry) lives here without dedicated schema columns. Existing rows without a field fall through to current default behavior.

### Automations and integrations

- Plain-English agent rules (`AgentRule` model, budget-metered NL compiler, conflict detection), category-scoped autopilot settings, snippets miner cron, scheduling sessions, automation run traces with rollback, and cron-driven workflow templates.
- Static-first rule evaluation (`lib/agent/static-rules.ts`): deterministic sender/domain/subject/body conditions on `AgentRule`/`SenderRule` run before any LLM classification (gate in `lib/agent/jobs.ts` via `tryStaticClassification` in `lib/agent/classify.ts`). A static match short-circuits with confidence 1 and zero AI budget spend; precedence is AgentRule over SenderRule, exact email over domain. Rules are versioned (`version` column, `agent_rule.version_snapshot` AuditLog snapshots on behavior-changing edits) and executions record the rule id/version that fired (AgentToolCall output, `agent_job.completed` audit payload).
- Rule dry-run preview (`POST /api/agent-rules/dry-run`): evaluates a saved rule or inline conditions against up to 200 recent conversations and reports matched/skipped, per-match evidence, and the planned action (attention, workflow status, Gmail labels, automation-level applicability) without mutating anything — the only writes are one `agent_rule.dry_run` audit row and the rule's `lastDryRunAt`. Manually-built static rules start as drafts and cannot be enabled until a dry-run has run (enforced in `PATCH /api/agent-rules/[id]`). Managed in the settings "Attention Rules" panel (create, preview, enable/disable, version history).
- `GET /api/cron/agent-jobs` executes pending `AgentJob`s (LLM classification, follow-up, lead-sequence) through `runAgentJob`: at most 25 jobs per run with per-tenant round-robin fairness, atomic pending → running claims so overlapping runs never double-execute, and per-job failure isolation. Pending jobs older than 7 days are bulk-failed as `stale_at_executor_launch` (200/run) rather than executed. Autopilot sends stay gated behind opt-in, learned profile, policy, budget, confidence/per-intent thresholds, daily cap, and failure limit — executing jobs does not enable sending. Auto-send additionally requires automation Level 5 (trust ladder above).
- Snooze persists a valid pre-snooze priority, marks snoozed conversations as `none`, and restores the saved priority on resurface with a medium fallback for legacy values.
- Google Calendar (events, free/busy, calendar holds) and optional MindBody connector. The Google Drive connector (OAuth foundation only, never injected into drafts) was removed 2026-07-11: `drive.readonly` is a Google *restricted* scope, and requesting it with no shipped feature would block the app's OAuth verification. The `GoogleDriveCredential` table and its admin-rekey handling remain so previously stored tokens stay intact if the connector returns (likely with the non-restricted `drive.file` scope).
- **Scheduling end-to-end** (`lib/agent/scheduling.ts`, `lib/agent/scheduling-booking.ts`): inbound scheduling requests create a `SchedulingSession`; the conversation Scheduling panel proposes free slots from the connected Google Calendar (`proposing`). When the counterparty replies agreeing to a slot, deterministic confirmation detection (`detectSchedulingConfirmation` — quoted history stripped, decline/counter-proposal guards, affirmative required, and the slot must be unambiguous: ordinal/weekday/time mention, or a bare yes only when a single slot was proposed; no LLM) moves the session to `confirmed` (audited `scheduling_session.confirmed`). Booking then creates the real calendar event via the existing integration — converting an active calendar hold at the agreed time in place (no duplicate event; a stale hold at another time is cancelled) and inviting the counterparty as attendee — and marks the session `booked` (audited `scheduling_session.booked`). Automated booking is trust-ladder-gated: `auto_book_event` requires Level 5 (an invite emails the counterparty — same outward tier as auto-send); below that the confirmation raises a pending `ApprovalRequest` (`step: "book_event"`, the unified primitive; audited `scheduling_session.booking_approval_requested`) and approving it — singly or in bulk — executes the booking. User-initiated booking (`POST /api/conversations/[id]/scheduling/book`, the panel's "Book calendar event" button) is never level-gated and cancels any pending book_event request. Calendar API failures never strand the session: the session stays `confirmed`, the error is recorded (`lastBookingError`/`lastBookingAttemptAt`), audited (`scheduling_session.booking_failed`), surfaced in the panel, and retryable there. All four scheduling audit actions appear in `/audit` and the per-conversation timeline.

## Query and performance constraints

- Mobile inbox list is paginated at 50 conversations per page (offset + "Load more" link). The `?page=N` param preserves all active filters.
- Desktop sidebar (`AppListColumn`) fetches the top 50 conversations per filter view. The needs-reply badge count is a direct `prisma.conversation.count()` against deterministic `stateRecord` columns — body/sender regex heuristics are not applied to the count (badge may be slightly high for fully unclassified inboxes).
- Status counts (groupBy) are cached once per tenant per 60 s in their own `unstable_cache` entry (`["app-list-counts", tenantId]`) and shared across all filter views. `mail/page.tsx` passes its already-fetched counts into `AppListColumn` to avoid a duplicate groupBy on desktop renders.
- Background jobs are bounded: `getStaleConversations` (`lib/agent/follow-up.ts`) processes at most 200 conversations per run; `close-fyi` admin route processes at most 100; the agent-jobs executor runs at most 25 jobs and stale-fails at most 200 per run.
- Home view (`commandCenterConversations`) is capped at `HOME_CONVERSATION_LIMIT = 25` conversations with `HOME_MESSAGE_LIMIT = 5` messages each.

## Important limitations

- Outlook archive/trash/read/draft writeback, category projection, feedback, and reconcile now have parity with Gmail (see Outlook section above); live Graph verification is still pending a dev Outlook credential.
- Gmail `cid:` inline images are only partially supported through size-capped embedding and still need broader safe rendering coverage.
- Classification heuristics still have edge-case overlap between command-center FYI logic and inbox list filtering.
- Learned `SenderRule` suggestions remain accept/dismiss only; manual creation/editing happens through static `AgentRule`s in the Attention Rules panel. Structured static rules do not yet get conflict detection, and the work-item-sync path evaluates rules by sender only (subject/body-conditioned rules apply on the agent-job classification path).
- Knowledge-base matching is keyword-based; crawling is single-page only.
- Team inboxes, roles, paid-plan enforcement, and additional integrations are not implemented.
